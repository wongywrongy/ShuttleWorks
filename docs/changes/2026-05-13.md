## Step 4 ‚Äî Supabase Auth

### Summary
Authentication scaffolding wired end-to-end. The backend grows a `get_current_user` dependency that verifies the JWT against Supabase and is attached to every data router; the frontend gets a Supabase client, an `AuthProvider`, a `LoginPage`, an `AuthGuard`, and an axios interceptor that attaches the JWT to every request. A `SUPABASE_URL=""` / `VITE_SUPABASE_URL=""` toggle keeps local dev + pytest working without a real Supabase project.

### Files added
- `backend/app/dependencies.py` ‚Äî `get_current_user(credentials=Depends(HTTPBearer(auto_error=False)))`. Lazy-constructs the Supabase client; returns a synthetic local-dev user when `SUPABASE_URL` is blank; maps missing / invalid tokens to HTTP 401.
- `tests/unit/test_dependencies.py` ‚Äî 7 tests covering both modes: synthetic-user when unconfigured, 401 on missing/invalid token, OK on valid token, missing-email tolerance, `user is None` from the provider.
- `frontend/.env.example` ‚Äî VITE_API_BASE_URL + VITE_SUPABASE_URL + VITE_SUPABASE_ANON_KEY.
- `frontend/src/lib/supabase.ts` ‚Äî exports `supabase: SupabaseClient | null` + `isAuthConfigured`.
- `frontend/src/context/AuthContext.tsx` ‚Äî `AuthProvider` + `useAuth()` hook. Subscribes to `supabase.auth.onAuthStateChange`; synthesises a session/user in local-dev mode.
- `frontend/src/app/AuthGuard.tsx` ‚Äî redirects to `/login` when no session; no-op in local-dev mode (synthetic session is truthy).
- `frontend/src/pages/LoginPage.tsx` ‚Äî email/password form + Google SSO button. Shows a banner in local-dev mode.

### Files modified
- `backend/requirements.txt` ‚Äî added `supabase>=2.0.0`.
- `backend/app/main.py` ‚Äî every data router now registers with `dependencies=[Depends(get_current_user)]`. `/health` and `/health/deep` stay public; Step 7's `/invites/:token` joins the public set when it lands.
- `frontend/package.json` ‚Äî added `@supabase/supabase-js` (2.105).
- `frontend/src/api/client.ts` ‚Äî axios request interceptor attaches `Authorization: Bearer <jwt>` from the current Supabase session. Reads at request time so token refreshes propagate without re-instantiating the client.
- `frontend/src/app/App.tsx` ‚Äî wraps the router in `<AuthProvider>`; `/login` is public; `/` and `/tournaments/:id/*` are wrapped in `<AuthGuard>`. `/display` stays public so a venue TV doesn't need a login (it already takes the tournament id as a query param).

### Decisions made during execution
- **`SUPABASE_URL=""` ‚Üí auth disabled** is treated as a first-class mode rather than an error. Why: the spec mandates that the test suite (`Write tests for the dependency (mock the Supabase client)`) keep working, but route tests for the rest of the suite don't have a Supabase project to point at. A hard requirement would force every contributor (and every CI run) to set up a real project just to land an unrelated bug fix. The mode is symmetric on both sides: backend returns a synthetic `local-dev` user, frontend exposes a synthetic session. Step 8 deployment sets the real values and production transparently flips into JWT verification.
- **Auth applied at `app.include_router(dep=...)` rather than per-route.** Keeps the dependency wiring in one file; the individual route handlers don't need to import the dependency. The downside ‚Äî route-level tests that build minimal apps don't exercise the auth dep ‚Äî is intentional: those tests focus on route logic, not auth wiring, and the dep is covered exhaustively in `test_dependencies.py`.
- **HTTPBearer `auto_error=False`** so unauthenticated requests reach the dependency body (where local-dev mode can serve them). FastAPI's default behaviour would 403 before our code ran.
- **Axios interceptor reads `supabase.auth.getSession()` at request time** rather than caching the token in a Zustand store. Supabase JS handles refresh internally; reading per-request guarantees we always send the latest `access_token` without an extra subscription.
- **Login page in local-dev mode** renders a clear banner explaining the bypass and a Continue button rather than rendering an inert form ‚Äî pretending auth works when it doesn't would be confusing during a dev session.

- `tsc -p tsconfig.app.json --noEmit` ‚Äî clean.

### Decisions made during execution
- **Role check has no bypass mode.** `get_current_user` skips JWT verification when `SUPABASE_URL` is blank, but `require_tournament_access` always runs against the DB. This keeps the role logic exercised in tests + local dev and means the production code path differs from local dev only in *who* the user is (synthetic vs Supabase), not *whether* the membership table is consulted.
- **Stable UUID for the local-dev user.** Used `00000000-0000-0000-0000-000000000000` so the membership table can store and retrieve rows for the synthetic identity without ad-hoc string handling. Tournaments created in local-dev stamp this id as both `owner_id` and the `tournament_members.user_id`.
- **Non-member tournament access returns 403, not 404.** Hides the existence distinction so an attacker can't probe ids via membership behavior. Test expectations adjusted from 404 ‚Üí 403 across the suite where the tournament is either missing or the caller isn't a member.
- **Pathless `/schedule`, `/schedule/repair`, `/schedule/warm-restart`, `/schedule/stream`, `/schedule/validate` keep no role check.** They're stateless compute that takes the full problem in the body and don't read or write stored state. Authentication is sufficient (Step 4's router-level `get_current_user` still applies via `app.main`).
- **PATCH on a tournament is operator-level, not owner-level.** The spec explicitly says "POST/PATCH write routes: operator"; this matches the intent that an operator can rename / archive a tournament they've been delegated to manage. DELETE stays owner-only as the destructive escape hatch.
- **POST `/tournaments` is open to any authenticated user, no role pre-check.** The role-check dependency needs a `tournament_id` from the path, which doesn't exist yet at create time. Any logged-in user can create a tournament; the creator becomes its owner.
- **`add_member` is upsert by design.** A second call with a different role overwrites ‚Äî the right semantics for Step 7's "accept invite" flow, where a viewer might be upgraded to operator without a separate set-role call.

### Decisions made during execution
- **Token == invite id.** No separate token column; the `InviteLink.id` UUID is the token. Keeps the migration surface unchanged and the lookup is a single `session.get` by primary key. Tokens are unguessable enough at 128 bits for a v1 sharing flow.
- **Resolve returns 404 only for genuinely missing tokens.** Revoked / expired invites still return 200 with `valid: false` so the frontend's "this link is no longer valid" message is the only thing the recipient sees ‚Äî they don't learn that the token was once real. (For brand-new typos, 404 surfaces a clear "invite not found" message.)
- **Accept never downgrades and never overwrites owner.** Role ordering `viewer < operator < owner`; accept only upgrades. The owner accepting a viewer-link invite to their own tournament stays owner; an operator accepting a viewer link stays operator; a viewer accepting an operator link is upgraded.
- **`InviteCreateDTO.role` is a `Literal["operator", "viewer"]`.** "owner" cannot be granted via invite ‚Äî the only way to become owner is to create the tournament, matching the spec's invariant that ownership is non-transferable through links.
- **Owner-only `DELETE /invites/{token}` uses a custom dep, not `require_tournament_access`.** The token doesn't include the tournament_id in the URL; the custom dep does the invite lookup + owner check together so the handler stays a one-liner.
- **Members list is viewer-level.** Any member can see who else is in the tournament. The spec doesn't gate this; surfacing the list to non-owners avoids "is X already a member?" confusion when an owner is about to send another invite.
- **Public resolve goes through the same axios instance.** The interceptor attaches a bearer token when one exists (when the session is loaded). The backend ignores the header on this route. Alternative was a second axios instance without the interceptor, but the duplication isn't worth saving one redundant header on one route.
- **Frontend "auth disabled" mode**: every user is the same local-dev sentinel, so generating + accepting an invite ends up being self-invite. Confusing in isolation but useful: the dev can exercise the full flow without standing up two Supabase users. Production behavior with real auth is what the spec actually tests.

- **`docs/changes/2026-05-13.md` keeps a note that values were originally committed.** Rewriting history to hide the prior leak would be dishonest and unaudit-able. The current revision in the changelog shows the redaction was intentional and dated.
- **Cloud-mode validator raises at import time of `app.config`.** A blank `SUPABASE_URL` in cloud mode would silently force every request to the synthetic local-dev user ‚Äî that's a high-severity auth bypass on a deployed host, exactly the kind of failure that should crash-loop the container rather than serve traffic. Local mode keeps the empty-string sentinel because pytest + the desktop app rely on it.
- **`*.env` pattern added even though no `*.env` files exist today.** Defense in depth ‚Äî keeps a future contributor from accidentally committing `production.env` or `staging.env` thinking it'll be ignored by the `.env` line alone.
- **`!.env.example` whitelist explicit.** None of the ignore patterns above it currently match `.env.example` (gitignore `*` doesn't cross dots), but the explicit `!` is a guardrail against a future patch that tightens the pattern set.
