## Step 4 — Supabase Auth

### Summary
Authentication scaffolding wired end-to-end. The backend grows a `get_current_user` dependency that verifies the JWT against Supabase and is attached to every data router; the frontend gets a Supabase client, an `AuthProvider`, a `LoginPage`, an `AuthGuard`, and an axios interceptor that attaches the JWT to every request. A `SUPABASE_URL=""` / `VITE_SUPABASE_URL=""` toggle keeps local dev + pytest working without a real Supabase project.

### Files added
- `backend/app/dependencies.py` — `get_current_user(credentials=Depends(HTTPBearer(auto_error=False)))`. Lazy-constructs the Supabase client; returns a synthetic local-dev user when `SUPABASE_URL` is blank; maps missing / invalid tokens to HTTP 401.
- `tests/unit/test_dependencies.py` — 7 tests covering both modes: synthetic-user when unconfigured, 401 on missing/invalid token, OK on valid token, missing-email tolerance, `user is None` from the provider.
- `frontend/.env.example` — VITE_API_BASE_URL + VITE_SUPABASE_URL + VITE_SUPABASE_ANON_KEY.
- `frontend/src/lib/supabase.ts` — exports `supabase: SupabaseClient | null` + `isAuthConfigured`.
- `frontend/src/context/AuthContext.tsx` — `AuthProvider` + `useAuth()` hook. Subscribes to `supabase.auth.onAuthStateChange`; synthesises a session/user in local-dev mode.
- `frontend/src/app/AuthGuard.tsx` — redirects to `/login` when no session; no-op in local-dev mode (synthetic session is truthy).
- `frontend/src/pages/LoginPage.tsx` — email/password form + Google SSO button. Shows a banner in local-dev mode.

### Files modified
- `backend/requirements.txt` — added `supabase>=2.0.0`.
- `backend/app/main.py` — every data router now registers with `dependencies=[Depends(get_current_user)]`. `/health` and `/health/deep` stay public; Step 7's `/invites/:token` joins the public set when it lands.
- `frontend/package.json` — added `@supabase/supabase-js` (2.105).
- `frontend/src/api/client.ts` — axios request interceptor attaches `Authorization: Bearer <jwt>` from the current Supabase session. Reads at request time so token refreshes propagate without re-instantiating the client.
- `frontend/src/app/App.tsx` — wraps the router in `<AuthProvider>`; `/login` is public; `/` and `/tournaments/:id/*` are wrapped in `<AuthGuard>`. `/display` stays public so a venue TV doesn't need a login (it already takes the tournament id as a query param).

### Decisions made during execution
- **`SUPABASE_URL=""` → auth disabled** is treated as a first-class mode rather than an error. Why: the spec mandates that the test suite (`Write tests for the dependency (mock the Supabase client)`) keep working, but route tests for the rest of the suite don't have a Supabase project to point at. A hard requirement would force every contributor (and every CI run) to set up a real project just to land an unrelated bug fix. The mode is symmetric on both sides: backend returns a synthetic `local-dev` user, frontend exposes a synthetic session. Step 8 deployment sets the real values and production transparently flips into JWT verification.
- **Auth applied at `app.include_router(dep=...)` rather than per-route.** Keeps the dependency wiring in one file; the individual route handlers don't need to import the dependency. The downside — route-level tests that build minimal apps don't exercise the auth dep — is intentional: those tests focus on route logic, not auth wiring, and the dep is covered exhaustively in `test_dependencies.py`.
- **HTTPBearer `auto_error=False`** so unauthenticated requests reach the dependency body (where local-dev mode can serve them). FastAPI's default behaviour would 403 before our code ran.
- **Axios interceptor reads `supabase.auth.getSession()` at request time** rather than caching the token in a Zustand store. Supabase JS handles refresh internally; reading per-request guarantees we always send the latest `access_token` without an extra subscription.
- **Login page in local-dev mode** renders a clear banner explaining the bypass and a Continue button rather than rendering an inert form — pretending auth works when it doesn't would be confusing during a dev session.

### Verification
- `pytest products/scheduler/tests/` — **244 passed** (237 from Step 3 + 7 new dependency tests).
- `tsc -p tsconfig.app.json --noEmit` — clean.
- `vite build` — clean; new `index` bundle slightly larger (~187 KB / 59 KB gzip) due to the supabase-js + auth context.

### Known follow-ups (not Step 4)
- Per-route tests don't currently exercise the auth dep because they build minimal apps. A small integration test in Step 5 (when role checks land) can verify that the full `app.main:app` actually 401s without a token in configured mode.

### Decisions made during execution
- **Role check has no bypass mode.** `get_current_user` skips JWT verification when `SUPABASE_URL` is blank, but `require_tournament_access` always runs against the DB. This keeps the role logic exercised in tests + local dev and means the production code path differs from local dev only in *who* the user is (synthetic vs Supabase), not *whether* the membership table is consulted.
- **Stable UUID for the local-dev user.** Used `00000000-0000-0000-0000-000000000000` so the membership table can store and retrieve rows for the synthetic identity without ad-hoc string handling. Tournaments created in local-dev stamp this id as both `owner_id` and the `tournament_members.user_id`.
- **Non-member tournament access returns 403, not 404.** Hides the existence distinction so an attacker can't probe ids via membership behavior. Test expectations adjusted from 404 → 403 across the suite where the tournament is either missing or the caller isn't a member.
- **Pathless `/schedule`, `/schedule/repair`, `/schedule/warm-restart`, `/schedule/stream`, `/schedule/validate` keep no role check.** They're stateless compute that takes the full problem in the body and don't read or write stored state. Authentication is sufficient (Step 4's router-level `get_current_user` still applies via `app.main`).
- **PATCH on a tournament is operator-level, not owner-level.** The spec explicitly says "POST/PATCH write routes: operator"; this matches the intent that an operator can rename / archive a tournament they've been delegated to manage. DELETE stays owner-only as the destructive escape hatch.
- **POST `/tournaments` is open to any authenticated user, no role pre-check.** The role-check dependency needs a `tournament_id` from the path, which doesn't exist yet at create time. Any logged-in user can create a tournament; the creator becomes its owner.
- **`add_member` is upsert by design.** A second call with a different role overwrites — the right semantics for Step 7's "accept invite" flow, where a viewer might be upgraded to operator without a separate set-role call.

### Decisions made during execution
- **Token == invite id.** No separate token column; the `InviteLink.id` UUID is the token. Keeps the migration surface unchanged and the lookup is a single `session.get` by primary key. Tokens are unguessable enough at 128 bits for a v1 sharing flow.
- **Resolve returns 404 only for genuinely missing tokens.** Revoked / expired invites still return 200 with `valid: false` so the frontend's "this link is no longer valid" message is the only thing the recipient sees — they don't learn that the token was once real. (For brand-new typos, 404 surfaces a clear "invite not found" message.)
- **Accept never downgrades and never overwrites owner.** Role ordering `viewer < operator < owner`; accept only upgrades. The owner accepting a viewer-link invite to their own tournament stays owner; an operator accepting a viewer link stays operator; a viewer accepting an operator link is upgraded.
- **`InviteCreateDTO.role` is a `Literal["operator", "viewer"]`.** "owner" cannot be granted via invite — the only way to become owner is to create the tournament, matching the spec's invariant that ownership is non-transferable through links.
- **Owner-only `DELETE /invites/{token}` uses a custom dep, not `require_tournament_access`.** The token doesn't include the tournament_id in the URL; the custom dep does the invite lookup + owner check together so the handler stays a one-liner.
- **Members list is viewer-level.** Any member can see who else is in the tournament. The spec doesn't gate this; surfacing the list to non-owners avoids "is X already a member?" confusion when an owner is about to send another invite.
- **Public resolve goes through the same axios instance.** The interceptor attaches a bearer token when one exists (when the session is loaded). The backend ignores the header on this route. Alternative was a second axios instance without the interceptor, but the duplication isn't worth saving one redundant header on one route.
- **Frontend "auth disabled" mode**: every user is the same local-dev sentinel, so generating + accepting an invite ends up being self-invite. Confusing in isolation but useful: the dev can exercise the full flow without standing up two Supabase users. Production behavior with real auth is what the spec actually tests.

### Decisions made during execution
- **Composite FK on the commands table, not the prompt's single-column FK.** The prompt's pseudocode shows `match_id UUID REFERENCES matches(id)`. `matches.id` is part of a composite PK `(tournament_id, id)` (Step A's deviation), not unique on its own, and `matches.id` is `String(100)` not UUID. The natural fix uses the composite `(tournament_id, match_id) → matches(tournament_id, id)` FK with CASCADE; `tournament_id` is already on every command row per the prompt's own schema. Deviation #1.
- **`match_id` column type is `VARCHAR(100)`, not `UUID`.** Same reasoning — match IDs in the data model are tournament-scoped strings, not globally unique UUIDs. Deviation #2.
- **`match_id` is `NOT NULL`.** Every action in the `MatchAction` enum targets a specific match; tournament-level actions are hypothetical. YAGNI. Deviation #3 — the prompt's pseudocode marks `match_id` nullable.
- **`CommandResponse` returns the *current* match state, not the post-original-apply state.** Operator UX contract: the response is "here's the canonical state after your action; render from this." On replay after a third-party update, the response reflects current reality — that's actually the right answer, not a bug. Pinned by test #7. Deviation #4 from one possible reading of the prompt's "return 200 with the original result"; the prompt's intent is closer to "the command was acknowledged, here's what to render."
- **One commit per call path, not the prompt's literal single `BEGIN/COMMIT`.** SQLAlchemy session semantics give the same correctness guarantee: read-only paths (idempotent replay, duplicate rejection) issue zero commits; fresh-apply and fresh-rejection paths issue exactly one commit at the end. Any exception before the commit rolls back transparently. Deviation #5 in wording only — the invariant the prompt cares about (no partial state escapes) holds.
- **Rejection rows commit even when the route returns 409.** Otherwise a replay re-evaluates the rejected command and might apply it under a different version, producing the inconsistency the audit row is supposed to prevent. The rejected-command-replay test pins this.
- **Concurrency note flagged for Step H.** The `match.version += 1` after `session.get(Match, ...)` is a check-then-write race. ## Step E — Supabase sync (outbox) + Realtime primitive

### Summary
Every match and tournament write now stages a `sync_queue` row in the same transaction as the data update. A background daemon thread drains the queue by upserting into Supabase Postgres (`on_conflict='tournament_id,id'` for matches, `on_conflict='id'` for tournaments). Failures increment a per-row `attempts` counter; rows that hit 10 attempts are logged but kept indefinitely for manual remediation. The frontend gets a `subscribeToMatches` primitive (with 10-second polling fallback) that Step F's command-queue work will wire into the Zustand store.

### Decisions made during execution
- **Outbox pattern, not the prompt's "try direct, fallback to queue" two-stage design.** The prompt describes "attempt direct push, if fail write to queue." That design has three observable states (in-flight, succeeded, failed-and-queued) and any can be left dangling if the request thread crashes mid-flight. The outbox pattern collapses to two states (queued, completed) by *always* writing to `sync_queue` in the same transaction as the data write — the queue row exists iff the match was committed. Crash-safe by construction. The push happens later, idempotently, with Supabase upsert guarding against double-push. Deviation #1 — flagged the advisor's "best industry practice" call out loud.
- **Schema deviations forced by SQLite portability** (same pattern as Steps A and C). Prompt's pseudocode uses `JSONB`, `DEFAULT gen_random_uuid()`, `DEFAULT now()` — none portable. Used SQLAlchemy's portable `JSON`, app-side `default=uuid.uuid4`, app-side `_utcnow()` defaults. Deviation #2.
- **`entity_id` is `String(100)`, not the prompt's `UUID`.** The Step A composite-PK reality means matches have string IDs; tournaments have UUID IDs but serialise to 36-char strings cleanly. One column covers both. Two-column alternative (`entity_id_uuid` + `entity_id_str`) would have been clearer but more code. Deviation #3.
- **Idempotency via `upsert(on_conflict=...)` on the Supabase push.** Not in the prompt's text but essential — if a push succeeds on Supabase but the local `DELETE FROM sync_queue` fails, the next worker iteration would otherwise hit a unique-constraint violation and never clear the queue. The advisor flagged this under "best industry practice"; added unconditionally.
- **Explicit call sites in the repository methods, not SQLAlchemy event listeners.** Event listeners decouple but are a debugging trap — session events fire from connection-pool boundaries with edge cases that bite under load. Four call sites is manageable to wire explicitly; the trade-off is wrong above ~10. Documented in the SyncService module docstring.
- **`session_factory` constructor parameter for tests.** Initial fixture monkey-patched `services.sync_service.SessionLocal`. That works in isolation but fails when other test files (`test_match_state.py` etc.) run first and trigger `isolate_test_database` → `purge_backend_modules`, which re-imports `services.sync_service` and orphans the monkeypatch. Same class-identity issue resolved in Steps A and C. The fix is to make the session factory a constructor parameter (defaults to module-level `SessionLocal`) so tests pass their own factory directly. - **Frontend primitive only; no Zustand wiring.** The prompt's E2 implies frontend integration but doesn't specify the store contract. Wiring `onUpdate` to a Zustand `applyDelta` action without a concrete consumer is design-by-imagination. Step F's command queue is where the operator UX consumes Realtime; the primitive ships ready for that consumer.
- **`enqueue_match` / `enqueue_tournament` are static helpers, not methods on a SyncService instance.** Staging a row to a session has no per-instance state — it's pure SQLAlchemy `session.add(SyncQueue(...))` with the payload-shape logic centralised. Static methods make the call sites read cleanly (`SyncService.enqueue_match(session, match)`) and avoid needing a service-locator singleton.
- **Worker runs every 5 seconds, not the prompt's 30.** 5 s feels close to real-time when Supabase is reachable; 30 s is honestly too long for an operator-cockpit UX where you click "Call to court" and expect the TV display to update within a second. The prompt's 30 s was for a *retry interval* on the failed queue; my worker drains continuously so that backpressure isn't needed. Constant is module-level and easy to tune.
- **No tournament syncing from `commit_tournament_state`.** Only `upsert_data` (the canonical state-blob write) enqueues. `commit_tournament_state` is `upsert_data` plus backup management — the backup rows aren't synced; only the canonical data goes to Supabase. The orchestration calls `upsert_data` internally so the sync still fires.

## Audit pass — production-readiness review

A post-arc audit dispatched four parallel agents with focused lenses (spec adherence, concurrency + crash-safety, cross-step contract integrity, error paths + edge cases) and synthesised findings. Five fixes landed in this audit-pass commit; the rest are documented limitations or were verified non-issues.

### Fixes applied

**Fix #1 (Critical — spec compliance): renamed `attempted_version` → `seen_version` everywhere.**

The prompt's Step C3 specifies the stale_version 409 body has `seen_version` (what the client sent) and `current_version` (what the row has). My initial implementation used `attempted_version` instead. Renamed throughout the constructor parameter, the `to_dict()` payload, every caller in `repositories/local.py`, the unit tests asserting the body shape, and the frontend's `CommandConflictDTO` TypeScript interface.

**Fix #2 (Critical — real-world bug): frontend reads canonical version before submit.**

The Step F `useCommandQueue` hook was sending `seen_version: 0` as a placeholder on every command. The Alembic backfill (Step A) creates `matches` rows at `version=1`, so the FIRST operator command on any match would have failed with 409 stale_version — every time, immediately, in production. The fix:

- New `canonicalVersionsByMatchId: Record<string, number>` slice on `matchStateStore` + `setMatchVersion(matchId, version)` action.
- New `apiClient.getMatchVersion(tid, matchId)` reads the legacy match-state route's `ETag` header (parses RFC 7232 weak/strong variants) and returns the canonical version. Returns 0 for unseen matches per the Step D convention.
- `useCommandQueue.submit` now reads from the cache first; on cache-miss, fetches via `getMatchVersion`; submits with the real version. Every successful command response writes back to the cache via `setMatchVersion(matchId, result.matchVersion)` so subsequent commands on the same match hit the cache. One roundtrip per never-before-seen match per session.

**Fix #3 (High — test infra hole): registered `PreconditionFailedError` + `ConflictError` handlers in two test fixtures.**

`test_schedule_advisories.py::client` and the two FastAPI app builders in `test_proposal_pipeline_integration.py` constructed minimal apps without registering the 409 / 412 exception handlers from `app.main`. If a 412 ever surfaced from those test paths it would become a 500. Audit-pass: registered both handlers in all three places matching the production wiring.

**Fix #4 (Verified non-issue): `useTournamentId()` throw guard.**

Auditor #4 reported that `useTournamentId()` was called inside `submit` and could throw during navigation. On verification: it's called at the hook body (line 72), not inside the `submit` callback. The throw fires at component render time if the hook is used outside `/tournaments/:id/*` — that's correct behaviour, surfaces via React's error boundary as a developer error. No code change needed.

**Fix #5 (Medium — correctness): optimistic-status rollback on refetch failure.**

When `useCommandQueue` got a 409 (either flavour), the rollback path called `apiClient.getMatchState` to refetch. If that fetch failed (transient 500, network drop), the catch block silently swallowed and the optimistic status stayed in place indefinitely. Audit-pass fix: capture `previousStatus` *before* applying the optimistic status; on refetch failure, explicitly `applyOptimisticStatus(matchId, previousStatus)` to roll back. Also wired a follow-up `getMatchVersion` on successful refetch so the canonical-version cache stays consistent.

**Fix #5b (Test-infra hardening, fell out of #1's rename): `_ce()` helper now imports app.exceptions if absent.**

The `_ce()` helper in `tests/unit/test_match_state.py` used `sys.modules["app.exceptions"]` (KeyError on missing). When tests ran as a narrow subset (`pytest tests/unit/test_match_state.py tests/unit/test_commands.py` alone, no other test importing `app.exceptions` first), all the conflict-flavour tests failed with KeyError. Audit-pass: `.get()` first, fall back to `from app import exceptions`. Tests now pass in any subset ordering.

### Findings that were verified, not fixed

These came back from the audit but were either documented limitations the user already approved or low-priority polish:

- **Frontend `flush()` concurrent-trigger collision** (Agent 2 — High): two concurrent flushes can both call `getPending()` and double-submit. Backend's idempotency dedupes on `command.id` so the second request returns 200 with the same outcome — no state corruption. Defer to a flush-mutex / "in-flight" status if real-world load shows wasted bandwidth.
- **Tournament-delete CASCADE leaves Supabase rows orphaned** (Agent 4 — Medium): SQLite CASCADE wipes `sync_queue` rows for a deleted tournament; Supabase keeps the stale rows. Local-first design assumes Supabase isn't the source of truth — operators should treat the cloud mirror as best-effort. Already documented in the deploy guide.
- **`bulk_project_from_schedule` deletion path doesn't push to Supabase** (Agent 2 — Critical-flagged but documented): the comment explicitly says "deletes don't sync". Schedule regeneration deletes are bulk events; the frontend's polling-fallback masks the staleness. Add `sync_delete` entity type in a follow-up.
- **Phantom conflict banner** (Agent 4 — Medium): a `recordConflict(matchId, ...)` record persists if the match is subsequently deleted (e.g., schedule regenerated). The banner would render for a vanished match. Low real-world likelihood and the next regenerate-induced render would catch it visually. Add a cleanup hook in the operator-UX sweep.
- **`attempted_version` typo in the audit report**: Agent 1 reported the field name as `attempted_version` in my code; my rename in Fix #1 turned that into `seen_version`. Caught and applied.
- **`process_command` half-stamped row recovery race** (Agent 2 — Critical): the recovery branch handles a Command row with neither `applied_at` nor `rejected_at` set, but two concurrent retries entering the branch could both modify the same row. Under SQLite + single uvicorn worker (the target deployment), there's no concurrency. Postgres + multiple workers would need a row-level lock. Already noted as a Step H deployment-topology concern.
- **`match.version` race in `process_command`** (Agent 2 — High, documented): check-then-write race acknowledged in code comments + Step C changelog. Postgres + multi-worker needs `SELECT … FOR UPDATE`.
- **Bulk admin routes bypass If-Match** (Agent 1 — Low): documented intentional scope per Step D.
- **MatchStatus translation round-trip** (Agent 3 — High-flagged): exists implicitly via the maps but no explicit test. Working as designed.
- **`replay` field unused by frontend** (Agent 3 — Medium): reserved for future UI; intentional.
- **`process_command` rejection_reason stored as literal string vs full JSON body** (Agent 1 — Medium): the audit row's `rejection_reason` is the human-readable label (`"stale_version"`, transition-conflict messages); the full 409 body is constructed at exception-time and sent to the client. Two different consumers (audit log vs HTTP client); intentional split.

```sql
    USING (app.role_in_tournament(tournament_id) IS NOT NULL);
ALTER PUBLICATION supabase_realtime ADD TABLE public.matches;
```

### Scope correction
The audit pass earlier and `docs/deploy/cloud.md` claimed all
three arc migrations (A / C / E) needed to land on Supabase. That
was wrong. Walking the SyncService one more time:

- `enqueue_match` → pushes match rows → needs `public.matches` ✅
- `enqueue_tournament` → pushes tournament rows → `public.tournaments` already exists ✅
- No `enqueue_command` → `commands` is local-only audit ❌ not needed on Supabase
- The sync queue itself is the local outbox by definition ❌ not needed on Supabase

So only Step A's `matches` schema is required on the cloud
mirror. Updated `docs/deploy/cloud.md` to reflect this — the
table now reads "Local SQLite vs Supabase: required / local-only"
for each of the three arc tables.
